# 83. 成绩发布审计流水

## 背景

`score_release` 只能表达当前成绩是否发布，不能回答“谁在什么时候发布或撤回过、影响了多少答卷、通知了多少学生”。真实考试系统需要可追溯流水，尤其在成绩撤回、学生申诉、管理员审计时不能只看当前状态。

## 本次改动

- 新增 `score_release_log` 表，记录成绩发布和撤回的历史动作。
- 发布成绩时写入 `PUBLISH` 流水，记录状态流转、操作者、完成答卷数、通知学生数、通知答卷数。
- 撤回成绩时写入 `REVOKE` 流水，记录撤回原因、撤回前可见答卷数、通知学生数、通知答卷数。
- 删除/清理考试数据时同步清理对应成绩发布流水。
- 新增接口 `GET /api/exams/{id}/score-release-logs`，管理员和教师按考试数据范围查询。
- 教师端考试管理增加“成绩记录”入口，使用抽屉展示发布/撤回历史。
- 质量门禁新增后端服务、控制器、前端 API、教师端入口和迁移表结构检查。

## 三端协同意义

- 教师端：能在发布/撤回之后查看完整历史，确认成绩是否通知到学生。
- 学生端：后续申诉和成绩不可见提示可关联最近一次撤回流水。
- 管理员端：后续可基于同一张流水表做全局成绩发布审计、异常告警和导出。

## 状态设计

- `action = PUBLISH`：成绩从未发布/已撤回进入已发布。
- `action = REVOKE`：成绩从已发布进入已撤回。
- `status_from` / `status_to` 保留数字状态，当前沿用 `0 = 未发布/已撤回`，`1 = 已发布`。
- `note` 在发布时可为空，撤回时为撤回原因。

## 验收点

- 新库和旧库启动后都存在 `score_release_log`。
- 发布成绩后新增一条 `PUBLISH` 流水。
- 撤回成绩后新增一条 `REVOKE` 流水，并保留撤回原因。
- 教师端“成绩记录”抽屉能看到动作、状态流转、处理人、影响答卷、通知数、时间和原因。
- 本地质量门禁通过。
