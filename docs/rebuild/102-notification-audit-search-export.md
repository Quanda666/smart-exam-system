# 102. 通知审计查询与导出

## 背景

在线考试系统的三端协同高度依赖站内通知：考试发布通知学生、成绩发布和撤回通知学生、监考提醒和强制交卷通知学生、成绩申诉处理通知学生。前面批次已经逐步让通知携带 `relatedType/relatedId`，但管理员端还缺少全局追踪能力。

没有通知审计时，出现“学生是否收到提醒”“成绩撤回是否通知到人”“申诉回复是否触达”等争议，只能查数据库。本批把通知发送记录纳入管理员系统日志工作台，形成可检索、可导出的协同证据。

## 本次改动

- 新增管理员通知审计查询接口：
  - `GET /api/notifications/audit`
- 新增管理员通知审计导出接口：
  - `GET /api/notifications/audit/export`
- 查询筛选条件包括：
  - `keyword`：匹配收件人、用户名、标题、内容、类型、关联类型。
  - `type`：通知类型。
  - `relatedType`：关联业务类型，例如 `EXAM_ATTEMPT`、`SCORE_APPEAL`。
  - `relatedId`：关联业务 ID。
  - `read`：已读/未读。
  - `userId`：接收人用户 ID。
  - `startFrom/startTo`：通知创建时间范围。
- 导出字段包括：
  - 通知 ID、接收人、用户名、用户 ID、类型、标题、内容、是否已读、关联类型、关联 ID、跳转链接、创建时间。
- 管理员 `SystemLog` 页面新增 `Notification Audit` 标签页。
- 本地质量门覆盖后端接口、服务层过滤导出、前端 API、管理员页面标签和导出函数。

## 三端协同

- 学生端继续接收通知，不增加额外操作负担。
- 教师端触发的考试发布、监考处置、成绩处理、申诉处理等动作可以通过通知审计追踪触达结果。
- 管理员端可以按业务对象追踪通知，例如按 `EXAM_ATTEMPT + attemptId` 追查某个答卷相关通知，或按 `SCORE_APPEAL + appealId` 追查申诉处理通知。
- 通知审计只证明系统生成了站内通知，不等同于学生已经阅读；已读状态单独展示。

## 验收点

- 管理员可以在系统日志页打开 `Notification Audit` 标签。
- 可以按关键字、类型、关联业务、已读状态和时间范围筛选通知。
- 可以导出当前筛选条件下的通知审计 CSV。
- 导出最多 5000 条，避免一次性导出拖垮系统。
- 非管理员不能访问通知审计接口。
- 通知审计不改变通知本身的已读状态。
