# Step 349 - Monitor Navigation Event Closure

## 背景

上一轮学生端已开始采集 `PAGE_UNLOAD_ATTEMPT`、`HISTORY_BACK_ATTEMPT`、`CONTEXT_MENU` 三类事件，但监考闭环仍不完整：后端白名单、风险权重、历史风险回填、监考面板筛选和展示文案都需要同步，否则会出现学生端已记录、教师端或管理员端无法筛选/理解的断点。

## 本次改动

- 后端 `MonitorService` 允许上报三类新增事件。
- 后端导出/展示文案补齐：
  - `PAGE_UNLOAD_ATTEMPT`：尝试离开页面
  - `HISTORY_BACK_ATTEMPT`：尝试返回上一页
  - `CONTEXT_MENU`：打开右键菜单
- 风险权重补齐：
  - 尝试离开页面按 5 分计入风险。
  - 尝试返回上一页按 4 分计入风险。
  - 打开右键菜单按 3 分计入风险。
- 数据库迁移器的历史风险回填 CASE 同步新增事件，避免老数据修复时分值不一致。
- 前端 `ExamMonitorPanel` 的事件筛选下拉和事件列表展示同步新增事件。
- 质量门新增结构性断言，防止后续只保留采集端而丢失后端接收、计分或面板展示。

## 三端协同意义

- 学生端：负责采集导航、离页、右键菜单等过程风险信号。
- 教师端：可以在实时监考面板中筛选这些事件，结合风险分进行人工判断和提醒。
- 管理员端：导出的监考事件与会话审计中具备可读事件名和统一风险分，便于追溯。

## 边界

这些事件仍然只作为风险记录，不自动判定作弊或违规；是否警告、备注、强制交卷仍由教师或管理员基于考试规则和上下文人工处理。

## 验收点

- 新事件可以通过 `/api/monitor/cheat-events/batch` 被接受。
- 监考事件导出中显示可读事件名。
- 监考会话风险分会累加新增事件分值。
- 监考面板事件筛选包含新增事件。
- 质量门覆盖后端白名单、风险权重、迁移回填和前端展示。
