# 第十八批落地：题目版本与审核日志

## 本批目标

在题目审核流和审核队列基础上，补齐题库治理的可追溯能力，并开始解决“题目审核通过后再次编辑会污染已组卷内容”的问题。

## 已落地内容

- `question` 新增 `version_no`，题目内容每次编辑递增版本号。
- 新增 `question_version` 表，保存题目每个内容版本的题干、答案、解析、分值、来源、审核态和可用态快照。
- 新增 `question_version_option` 表，保存每个题目版本对应的选项快照。
- 新增 `question_review_log` 表，记录创建、编辑、提交审核、审核通过、审核驳回、上架、下架、删除等动作。
- `paper_question` 新增 `question_version_id`，试卷选题时固定引用当时的题目版本。
- 发布考试生成题目快照时优先读取 `paper_question.question_version_id` 对应的版本内容和选项，避免后续题目编辑影响已组卷并发布的考试。
- 题库列表返回 `versionNo`，页面在审核列展示当前版本号。
- 题库页面新增单题“日志”抽屉，可查看题目审核/版本操作流水。
- 修复教师审核待审题的可见性边界：教师可在审核队列看到待审题，但仍不能审核自己创建的题目。

## 数据迁移策略

- 旧库启动时自动为 `question` 补 `version_no`，默认为 `1`。
- 自动创建题目版本表、版本选项表和审核日志表。
- 自动为历史题目生成当前版本快照。
- 自动为历史试卷题目回填 `question_version_id`，指向题目当前版本。

## 后续待做

- 试卷发布态也需要状态机保护：已发布试卷不应被直接删除重建题目明细。
- 题目版本需要更完整的版本对比视图，支持查看两个版本的题干、答案、选项差异。
- 审核日志应接入统一操作日志/审计中心，支持管理员按操作者、时间、动作类型检索。
- 后续应增加 `role_action_permission`，把审核通过/驳回从硬编码角色升级为可配置动作权限。
