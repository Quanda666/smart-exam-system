# 82. 成绩发布审计字段拆分

## 背景

成绩发布和撤回原先共用 `score_release.note`，教师端也用这个字段判断是否存在撤回记录。这个设计会把“发布说明”和“撤回原因”混在一起，后续做管理员审计、成绩撤回复盘、通知追踪时语义不够可靠。

## 本次改动

- `score_release` 新增 `publish_note` 和 `revoke_reason`。
- 旧库启动迁移时自动补列，并将历史 `note` 回填到 `revoke_reason`。
- 发布成绩时清空撤回人、撤回时间、撤回原因和兼容 `note`。
- 撤回成绩时写入 `revoke_reason`，同时写入旧 `note` 作为兼容兜底。
- 教师端考试列表新增 `scorePublishNote`、`scoreRevokeReason` 字段。
- 教师端成绩发布状态提示优先读取独立字段，不再只依赖 `scoreReleaseNote`。
- 质量门禁增加对独立审计字段、迁移和前端展示逻辑的静态检查。

## 三端协同意义

- 教师端：撤回成绩必须留下明确原因，列表中可直接看到撤回人、撤回时间和撤回原因。
- 学生端：成绩撤回通知仍按 attempt 关联，后续可用同一原因解释成绩暂不可见。
- 管理员端：后续可以基于 `publish_note` / `revoke_reason` 做统一审计页、导出和异常告警，而不会误读旧 `note`。

## 兼容策略

- 保留 `score_release.note`，避免旧数据和旧接口立即失效。
- 查询字段使用 `COALESCE(new_column, note)`，历史记录仍可展示。
- 新写入路径开始使用明确字段，逐步把旧字段降级为兼容字段。

## 验收点

- 新数据库 `schema.sql` 包含 `publish_note`、`revoke_reason`。
- 旧数据库启动后自动补齐两个字段，并将旧 `note` 回填到 `revoke_reason`。
- 发布成绩后学生可见成绩，撤回审计被清空。
- 撤回成绩后学生不可见成绩，教师端可见撤回人、时间、原因。
- 静态质量门禁通过。
