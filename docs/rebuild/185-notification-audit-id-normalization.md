# 185. 通知审计 ID 输入规范化

## 背景

教师端已经可以复制通知审计 ID，但实际协作中管理员收到的内容可能是 `123`、`Notification #123`、`ID: 123` 等不同格式。管理员通知审计接口按 `Long notificationId` 精确查询，如果前端直接提交带文字的内容，容易触发参数解析失败。

## 本批改动

- 管理员端 `Notification Audit` 的 `Notification ID` 输入框在失焦时自动规范化。
- 点击搜索前会规范化 ID，并将输入框同步为纯数字 ID。
- 点击导出前会使用同一套规范化逻辑，保证导出与列表查询一致。
- 规范化规则：从输入内容中提取第一段数字；没有数字时清空 ID 条件，避免向后端提交非法 `Long` 参数。
- `Notification Audit` 页签的顶部说明改为通知投递审计语义，避免误显示成绩申诉说明。
- 质量门禁增加失焦处理、规范化函数和查询参数断言。

## 三端协同影响

- 教师端：可以继续复制或转述 `Notification #id`。
- 管理员端：可直接粘贴教师提供的通知引用，系统自动提取可查询 ID。
- 学生端：不改变通知送达与已读状态，只增强事后审计定位效率。

## 验收点

- 输入 `Notification #123` 后失焦，输入框变为 `123`。
- 输入 `ID: 123` 后点击搜索，查询参数为 `notificationId=123`。
- 导出通知审计时使用同样的规范化 ID。
- 纯数字 ID 的既有查询行为不变。
