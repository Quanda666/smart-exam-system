# 158. 成绩申诉考试快照一致性修复

## 背景

第 156 批为成绩申诉增加 `exam_id` 快照，第 157 批兼容了历史空快照。继续审计后发现还需要处理一种迁移期边界：历史数据里如果 `score_appeal.exam_id` 不为空但与 `exam_attempt.exam_id` 不一致，成绩发布阻断可能错误统计到别的考试，或者漏掉真正所属考试。

本批把一致性规则明确下来：

- 能通过 `attempt_id` 找到答卷时，以 `exam_attempt.exam_id` 为准。
- 找不到答卷的孤儿申诉，才退回使用 `score_appeal.exam_id` 快照。
- 启动迁移会修复空快照和错快照。

## 本批改动

- 数据库迁移：`score_appeal.exam_id` 为空或与答卷考试不一致时，统一回填为 `exam_attempt.exam_id`。
- 教师端考试列表：申诉阻断计数有答卷时按答卷所属考试统计，孤儿申诉按快照统计。
- 发布成绩前检查：待处理申诉和待复核申诉使用相同优先级，避免错快照污染发布状态。
- 质量门禁：固定迁移必须修正不一致快照，固定发布阻断必须显式区分“有答卷”和“孤儿申诉”。

## 三端协同影响

- 学生端：无交互变化，提交申诉仍由后端写入考试快照。
- 教师端：考试列表的待申诉/待复核阻断原因更可信，历史错数据不会把阻断挂到错误考试上。
- 管理员端：后续全局审计、异常清理和数据质量报表可以基于这条优先级解释申诉归属。

## 验收点

- `score_appeal.exam_id IS NULL` 的历史行会被回填。
- `score_appeal.exam_id <> exam_attempt.exam_id` 的历史行会被修正。
- 有答卷的申诉按答卷考试阻断成绩发布。
- 没有答卷的孤儿申诉按申诉快照阻断成绩发布。
- 待处理申诉和待复核申诉都遵守同一归属规则。
