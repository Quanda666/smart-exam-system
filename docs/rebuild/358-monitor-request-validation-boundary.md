# Step 358 - Monitor Request Validation Boundary

## 背景

监考链路已经具备学生事件批量上报、教师监控会话、处置动作、强制交卷和审计导出。第 357 步收紧了核心 ID 边界，本次继续补请求体边界，避免无效请求进入后续状态判断或数据库查询。

正式在线考试系统不能只依赖前端表单或 Controller 层的默认行为。批量事件中的每一条事件都必须被递归校验，教师处置动作也必须在服务层明确拒绝空请求对象。

## 本次改动

- `CheatEventBatchRequest.events` 显式声明为 `List<@Valid CheatEventRequest>`。
- 保留批量事件外层约束：不能为空、单次最多 100 条。
- `MonitorService.createMonitorAction` 在查监考会话前调用 `requireMonitorActionRequest`。
- 新增 `requireMonitorActionRequest`，拒绝空的教师监考处置请求。
- 质量门禁新增结构断言，确保批量事件元素级校验和处置动作请求校验持续存在。

## 协同意义

- 学生端：批量上报时每条监考事件都必须满足 `attemptId`、事件类型、客户端事件 ID 等约束。
- 教师端：提醒、规则提醒、备注、强制交卷动作不会因空请求体进入会话权限查询或状态流转。
- 管理员端：监考审计只围绕结构完整的事件和处置动作展开，减少无效请求噪声。

## 验收点

- 批量事件请求为空数组时被拒绝。
- 批量事件超过 100 条时被拒绝。
- 批量事件中的单条事件缺少必填字段或 `attemptId <= 0` 时被拒绝。
- 空的监考动作请求不会进入监考会话查询。
- 合法批量事件、提醒学生、规则提醒、备注和强制交卷行为不变。
