# Step 356 - Score Appeal Appeal ID Boundary

## 背景

成绩申诉已经形成学生提交、教师回复、复核关闭、学生查看日志、教师导出日志、管理员统一审计的闭环。上一阶段收紧了学生提交时的 `attemptId` 和 `questionId`，但申诉生命周期后续入口仍以 `appealId` 作为核心路径参数。

如果 `appealId <= 0` 继续进入权限查询或精确筛选，会把参数错误伪装成“申诉不存在”“无权限”或全量列表回退。正式考试系统需要在服务层把这类非法 ID 明确挡住，避免页面绕过、通知深链异常和内部调用造成不一致行为。

## 本次改动

- `ScoreAppealService` 新增 `requirePositiveAppealId`，拒绝 `appealId == null` 或 `appealId <= 0`。
- 教师端申诉日志、日志导出、申诉回复、复核关闭统一通过访问检查在查库前校验 `appealId`。
- 学生端个人申诉日志和日志导出也复用同一正数边界。
- 教师端精确申诉筛选 `listAppeals(..., appealId, ...)` 在收到非空 `appealId` 时必须为正数，不再把非法值静默忽略。
- 质量门禁新增结构断言，防止后续重构绕过 `appealId` 服务层边界。

## 协同意义

- 学生端：个人申诉日志和导出只围绕有效申诉展开，非法链接会稳定返回参数错误。
- 教师端：通知深链、回复、复核关闭和日志导出共享同一 ID 边界，减少“查不到”和“无权限”混杂的排查成本。
- 管理员端：全局申诉审计仍只记录真实业务生命周期，非法 ID 探测不会产生额外审计噪声。

## 验收点

- `appealId <= 0` 不会进入 `score_appeal` 权限查询。
- 教师端日志、导出、回复、复核关闭都共享该边界。
- 学生端个人日志和导出也共享该边界。
- `GET /api/reviews/appeals?appealId=<id>` 对非法非空 ID 明确失败，对合法 ID 仍按授课范围过滤。
- 既有成绩发布、申诉窗口、复核待阅、通知深链和审计日志行为不变。
