# 第十七批落地：题目审核队列与行级动作权限

## 本批目标

在第十六批题目审核流基础上，补齐教师和管理员实际处理审核任务时需要的“待审队列”能力，并减少前端靠角色猜测操作权限的问题。

## 已落地内容

- 题目列表接口新增 `reviewStatus` 查询参数，支持按 `DRAFT`、`PENDING`、`APPROVED`、`REJECTED` 筛选题目。
- 后端题目列表为每条题目返回行级动作标记：
  - `canEdit`
  - `canDelete`
  - `canSubmitReview`
  - `canReview`
  - `canTakeOffline`
- 行级动作标记按当前登录用户、题目创建人、审核状态和可用状态计算，前端不再只靠角色判断审核按钮。
- 题库页面新增“审核状态”筛选器，可直接筛选草稿、待审核、已通过、已驳回题目。
- 题库页面新增“待审队列”入口，一键切到 `status=0` 且 `reviewStatus=PENDING` 的题目列表。
- 操作列改用后端返回的动作标记控制“编辑、提交审核、通过、驳回、下架、删除”按钮展示。
- 批量提交审核只处理后端判定可提交的题目；批量下架只处理可下架题目。

## 协同价值

- 管理员端：可以从题库直接进入待审队列，处理全局待审题目。
- 教师端：可以提交自己创建的题目；非创建人的教师在后端允许范围内可审核待审题目，创建人不能自审。
- 组卷链路：只有审核通过且可用的题目才能进入试卷，审核队列的筛选能力让题库治理更可操作。

## 后续待做

- 建立独立的动作权限表，例如 `role_action_permission`，将 `QUESTION_REVIEW_APPROVE`、`QUESTION_REVIEW_REJECT` 从角色硬编码升级为可配置授权。
- 建立独立审核工作台页面，支持只看“我可审核”的题目、按学科/来源/创建人聚合，以及批量通过/驳回。
- 建立题目版本表，审核通过后的题目被编辑时自动生成新版本，历史考试和已发布试卷继续引用旧版本。
- 增加审核操作日志，记录提交、通过、驳回、下架的操作者、时间、前后状态和意见。
