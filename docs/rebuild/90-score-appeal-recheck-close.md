# 90. 成绩申诉复核完成闭环

## 背景

第 87-89 批让成绩申诉有结构化处理结果，并让教师首页能直达“需要复核”的申诉。但 `RECHECK_REQUIRED` 只会形成待办，缺少“复核完成”动作，待办无法自然关闭。

## 本次改动

- `score_appeal` 新增复核完成字段：
  - `recheck_note`
  - `rechecked_by`
  - `rechecked_at`
- 新库 `schema.sql` 直接包含复核字段。
- 旧库启动迁移自动补列。
- 教师/管理员端新增接口：
  - `POST /api/reviews/appeals/{id}/recheck/close`
- 服务端状态约束：
  - 只有 `status = 1` 且 `handling_result = RECHECK_REQUIRED` 的申诉可以完成复核。
  - 完成复核后 `status` 更新为 `2`。
  - 记录复核说明、复核人、复核时间。
  - 向学生发送申诉复核完成通知。
- 教师端申诉列表新增“完成复核”动作和复核时间展示。
- 学生端申诉记录新增复核说明展示。

## 状态边界

- 本批次仍不自动改分。
- 如果复核结果涉及分数调整，应继续通过后续“成绩撤回/重算/重新发布”流程处理。
- `status = 2` 表示申诉复核待办已关闭，不代表考试成绩被自动修改。

## 验收点

- 普通已回复申诉不能调用复核完成接口。
- 待处理申诉不能调用复核完成接口。
- 只有 `RECHECK_REQUIRED` 的已回复申诉可以完成复核。
- 完成复核后教师首页“需复核申诉”计数减少。
- 学生端能看到复核说明。
- 本地质量门禁通过。
