# 30. 多实例调度锁

## 目标
- 解决后端横向部署多个实例时，自动审批提醒任务可能被多个节点同时触发的问题。
- 保持单体架构不变，先用数据库锁完成低成本的多实例互斥。
- 为后续 Redis 分布式锁或专用调度平台预留清晰边界。

## 后端改造
- 新增 `system_job_lock` 表：
  - `lock_key`：任务锁唯一标识。
  - `owner_id`：当前持锁节点。
  - `locked_until`：锁租约过期时间。
  - `acquired_at`：本轮锁获取时间。
  - `updated_at`：锁刷新/释放时间。
- 新增 `JobLockService`：
  - `tryAcquire(lockKey, leaseSeconds)`：抢锁。
  - `release(lockKey)`：释放当前节点持有的锁。
  - `ownerId()`：返回当前应用节点标识。
- `ApprovalReminderScheduler` 接入锁：
  - 锁名：`approval-reminder-scheduler`。
  - 租约：`600` 秒。
  - 抢不到锁时本节点跳过本轮任务。
  - 抢到锁后调用审批提醒服务，最终释放锁。

## 审计增强
- `exam_approval_reminder_log` 新增：
  - `node_id`：执行节点标识。
  - `duration_ms`：本次提醒处理耗时。
- 审批提醒日志接口返回：
  - `nodeId`
  - `durationMs`
- 审批队列页“提醒记录”新增“节点/耗时”列。

## 设计说明
- 锁使用数据库唯一键和过期时间实现：
  - 首次执行使用 `INSERT IGNORE` 初始化锁。
  - 后续只有锁已过期或当前节点仍为 owner 时才能更新租约。
  - 任务结束主动释放锁。
- 锁只保护调度入口，不改变手动提醒按钮：
  - 管理员手动提醒仍受冷却规则保护。
  - 自动任务同时受任务锁、任务间隔和提醒冷却三层限制。
- 抢不到锁不写审批提醒日志：
  - 这是节点运行层面的互斥结果，不代表审批提醒业务被处理。
  - 需要排查时可查看应用日志中的 skipped debug 信息。

## 三端协同影响
- 管理员端：
  - 能从提醒记录看到自动任务由哪个节点执行、耗时多少。
  - 减少重复提醒通知和重复审计记录。
- 教师端：
  - 审批 SLA 提醒更稳定，不因多节点部署产生通知噪音。
- 学生端：
  - 发布审批流程更稳定，降低临近开考仍未发布的概率。

## 风险与后续
- 当前数据库锁适合学院级并发和模块化单体阶段。
- 如果后续部署到多机房或高频任务场景，应升级为 Redis/Redisson 锁或独立调度平台。
- Batch 31 建议转向考试作答侧高可用基础：
  - 草稿保存幂等键。
  - 作答心跳与 Redis 草稿缓存设计。
  - 提交链路防重复与恢复策略。
